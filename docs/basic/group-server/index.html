群组服入门

约 8361 字大约 28 分钟

跨服多服务器核心

2026-02-05

想要搭建像嗨皮咳嗽（Hypixel）那样的小游戏服务器吗？想知道嗨皮咳嗽是怎么做到如此强大的分区管理的吗？想知道为什么这种服务器的性能为何如此优秀吗？管你想不想，给👴想就对了

如果要只做这样一个小游戏服务器，那么这里我们就需要了解到一个全新的概念——群组服。

什么是群组服？

基础概念

群组服（也称为跨服端、代理服务器、BungeeCord网络）是一种Minecraft服务器架构，它允许多个独立的Minecraft服务器（子服）连接在一起，玩家可以在这些服务器之间无缝切换，就像在一个大型服务器网络中游玩。

抽象比喻

如果不懂，就可以试着想象一下某个大型购物中心：

  • 群组服（代理端） = 购物中心的总入口和导航系统
  • 子服务器 = 购物中心里的各个商店（游戏大厅、生存服、小游戏服等）
  • 玩家 = 顾客，从总入口进入后可以自由进出各个商店

玩家只需要连接一个IP地址（代理服务器），就能访问整个服务器网络中的所有子服。

基本结构示意图

玩家 (你的客户端)
       ↓
   连接到同一个IP
       ↓
[代理服务器 - BungeeCord/Velocity]  ← 这是玩家看到的唯一入口
       ↓
  分发到不同子服
       ↓
   ┌───┴───┬───────┬───────┐
   ↓       ↓       ↓       ↓
[大厅服] [生存服] [小游戏] [创造服]
(lobby)  (survival) (games) (creative)

为什么需要群组服？

单服务器的局限性

这不废话吗？肯定是单服务器不够用啦~

单服务器就是我们所熟知的那些服务器核心（服务端），例如Spigot、Paper、Purpur等核心，而传统的单一Minecraft服务器则面临以下问题：

性能瓶颈

所有玩家、所有世界都被挤在一个服务器进程中，这就导致很容易变得卡顿

管理困难

来自四面八方、鱼龙混杂的玩法功能插件、模组被混合在了一起，而你又无法将它们各个独立拆分开来，所以想要管理这些玩法功能就会变得十分的困难

维护风险

如果你重启服务器，就会导致所有玩家跟着你一起掉线，有被问候祖宗十八代的风险

扩展受限

在单服务器里，你还想要“灵活”两个字？不存在的事情，单服务器是无法灵活添加新的游戏模式滴~

群组服的优势

相比之下，群组服在单服务器之上就有很大的提升，而且还有很大一部分的进步空间：

性能分散

每个子服独立运行，互不影响。比如小游戏服出现卡顿的时候就不会影响生存服

你还可以给不同子服分配不同的硬件资源

灵活管理

  • 可以单独重启某个子服而不影响其他服务器
  • 不同子服可以运行不同的Minecraft版本
  • 便于测试新插件（在测试服测试后再部署到正式服）

玩法多样化

这是群组服的一大优势，你可以在同一个服务器网络中提供：

  • 生存服（Survival）
  • 小游戏服（起床战争、空岛战争等）
  • 创造服（Creative）
  • 练习服（Practice）
  • 活动服（Events）

玩家体验提升

  • 玩家无需记住多个IP地址
  • 切换服务器时不需要断开连接重连
  • 跨服聊天、好友系统等社交功能

也就是说，你的玩家甚至还能待在一个私密的子服内干你想干嘛的事情……😊

商业价值

这是群组服的最大优势，群组服相比单服务器之下：

  • 更容易建立品牌（一个统一的服务器网络嘛，这个没得说）
  • 更容易运营活动（跨服排行榜、全服公告等等）
  • 能提高玩家留存率（可以提供多种玩法而满足不同需求）

群组服的工作原理

技术架构

群组服跟单服务器的区别也就在这里了：它们的技术架构有很大不同

三层结构

【玩家层】
   ↓ (Minecraft协议)
【代理层】BungeeCord/Velocity
   ↓ (修改过的Minecraft协议)
【服务器层】Spigot/Paper 子服务器

工作流程

  1. 玩家连接

    玩家使用客户端连接到代理服务器的IP，代理服务器接收连接请求

  2. 身份验证

    代理服务器验证玩家账号（正版验证或离线模式），验证通过后玩家进入网络

  3. 分配服务器

    根据配置，将玩家传送到默认服务器（通常是大厅），玩家数据就在代理层进行管理

  4. 服务器切换

    玩家使用命令或点击菜单进行切换服务器的操作后，代理服务器就会处理切换逻辑，将玩家连接转移到目标子服。切换过程中玩家不会断开连接

  5. 数据传输

    玩家的所有数据包经过代理服务器中转，代理可以拦截、修改或转发这些数据包，这样就能实现跨服聊天、权限同步等功能

重要概念 重要

记住这两个关键要点：

IP绑定模式（IP Forwarding）

由于所有玩家都是从代理服务器连接到子服，子服务器看到的玩家IP都是代理服务器的IP。为了获取真实玩家IP，需要启用IP转发：

  • 代理端：将真实IP信息附加到连接数据中
  • 子服端：读取并解析这些信息
  • 安全性：必须正确配置，否则会有安全风险

在线模式（Online Mode）

  • 代理服务器设置：online-mode: true（验证正版）或 false（离线模式）
  • 子服务器设置：必须设置为 online-mode: false
  • 原因：由代理服务器统一验证玩家身份，子服信任代理的验证结果

主流群组服

1. BungeeCord

开发者：SpigotMC团队
发布时间：2012年
当前状态：成熟稳定，使用最广泛

特点

历史最悠久，没有之一，生态也是最完善的，支持的插件也多，相关教程也丰富

适用场景

  • 中小型服务器网络
  • 需要大量插件支持的服务器
  • 预算有限的服主

版本支持

支持 Minecraft 1.7.x 至最新版本，同一网络中子服可以运行不同版本

2. Velocity

开发者：PaperMC团队
发布时间：2018年
当前状态：现代化、高性能的替代方案

特点

现代化设计，性能更好，而且有更好的安全性、更灵活的插件API和更少的内存占用

适用场景

  • 大型服务器网络（500+玩家）
  • 追求极致性能的服务器
  • 新建服务器（没有历史包袱）

版本支持

支持 Minecraft 1.7.2 至最新版本，对新版本支持更快

3. Waterfall 停更

基于：BungeeCord
开发者：PaperMC团队
当前状态：BungeeCord的优化分支，但已停止更新支持

特点

完全兼容BungeeCord插件，且性能优化好，漏洞修复更及时

适用场景

  • 想要BungeeCord兼容性 + 更好性能
  • 从BungeeCord迁移的中间选择

版本支持

已被PaperMC团队弃坑，不再更新支持更新的Minecraft版本

怎么选？

大部分服主都有一种选择困难症，尤其是对于这种不熟悉的领域。不要看了，说的就是你！

如果实在是不知道怎么选，就按照下面的选择流程指导来选：

选择BungeeCord，如果你：

  • 第一次搭建群组服
  • 需要使用大量现成插件
  • 服务器规模较小（<200人，不用想都知道你是这种）

选择Waterfall，如果你：

  • 想要更好的性能但保持兼容性
  • 从BungeeCord升级

选择Velocity，如果你：

  • 追求最佳性能
  • 服务器规模较大
  • 愿意学习新的插件开发方式

核心配置文件：config.yml

完整配置示例与详解

# 监听器配置 - 定义代理服务器监听的端口和行为
listeners:
- query_port: 25577              # Query协议端口（服务器列表查询）
  motd: '&6我的群组服务器'        # 服务器列表显示的MOTD（支持颜色代码）
  priorities:                    # 服务器优先级列表
  - lobby                        # 玩家登录后默认进入的服务器
  bind_local_address: true       # 绑定到本地地址
  ping_passthrough: false        # 是否将ping请求转发给子服
  query_enabled: false           # 是否启用Query协议
  proxy_protocol: false          # 是否启用Proxy Protocol
  forced_hosts:                  # 强制域名绑定（多域名指向不同服务器）
    pvp.example.com: pvp         # 当玩家使用pvp.example.com连接时，进入pvp服务器
    lobby.example.com: lobby
  tab_size: 60                   # Tab列表最大玩家数
  host: 0.0.0.0:25565           # 监听地址和端口
  max_players: 100              # 最大玩家数
  tab_list: GLOBAL_PING         # Tab列表模式：GLOBAL_PING, SERVER, GLOBAL
  force_default_server: false   # 是否强制玩家进入默认服务器

# IP转发设置 - 非常重要的安全配置
ip_forward: true                # 是否启用IP转发（必须与子服配置匹配）

# 权限设置
groups:
  admin:                        # 管理员组
    - bungeecord.command.alert
    - bungeecord.command.end
    - bungeecord.command.ip
    - bungeecord.command.reload
  default:                      # 默认玩家组
    - bungeecord.command.server
    - bungeecord.command.list

# 远程控制台（RCon）设置
remote_console:
  enabled: false                # 是否启用RCon
  password: "your_password"     # RCon密码

# 网络设置
network_compression_threshold: 256  # 数据包压缩阈值（字节）
timeout: 30000                      # 连接超时时间（毫秒）
connection_throttle: 4000          # 连接节流时间（防止频繁连接）
connection_throttle_limit: 3       # 节流时间内最大连接次数

# 玩家数据设置
player_limit: -1                   # 全局玩家限制（-1为无限制）
prevent_proxy_connections: false   # 是否阻止通过代理连接的玩家

# 在线模式 - 正版验证
online_mode: true                  # true=正版验证，false=离线模式（盗版）

# 日志设置
log_commands: false               # 是否记录玩家命令
log_pings: true                   # 是否记录ping请求

# 服务器定义 - 定义所有子服务器
servers:
  lobby:                          # 服务器名称（唯一标识）
    motd: '&a欢迎来到大厅'        # 服务器描述
    address: localhost:25566      # 服务器地址（IP:端口）
    restricted: false             # 是否限制访问（需要权限）
  
  survival:
    motd: '&2生存服务器'
    address: localhost:25567
    restricted: false
  
  creative:
    motd: '&e创造服务器'
    address: localhost:25568
    restricted: false
  
  minigames:
    motd: '&c小游戏服务器'
    address: localhost:25569
    restricted: false

# 禁用的命令列表
disabled_commands:
- disabledcommandhere

# 统计数据（是否发送匿名统计）
stats: true

# Forge支持
forge_support: false

# 注入命令（自定义命令别名）
aliases:
  server: "servers"  # 将/server命令别名为/servers

重要配置说明 重要

1. online_mode（在线模式）

# 正版服务器
online_mode: true

# 离线服务器
online_mode: false

这段不是废话，即使你知道online_mode的作用，你也需要注意：

  • 代理端：按需设置，别乱设置
  • 所有子服：必须设置 online-mode=false（在server.properties中）

2. ip_forward（IP转发）

# 代理端 config.yml
ip_forward: true
# 子服端 server.properties
online-mode=false

# 子服端 spigot.yml 或 paper.yml
bungeecord: true

3. servers（服务器列表）

servers:
  服务器ID:
    address: IP:端口      # 子服的实际地址
    motd: 描述            # 服务器描述
    restricted: false     # 是否需要权限才能进入

启动脚本

Windows启动脚本

start.bat
@echo off
title BungeeCord代理服务器
java -Xms512M -Xmx1G -XX:+UseG1GC -jar BungeeCord.jar
pause

Linux启动脚本

start.sh
#!/bin/bash
screen -dmS bungeecord java -Xms512M -Xmx1G -XX:+UseG1GC -jar BungeeCord.jar nogui
echo "BungeeCord已在screen会话中启动"
echo "使用 'screen -r bungeecord' 查看控制台"

使用方法：

chmod +x start.sh
./start.sh

下载与安装

  1. 下载Velocity

    访问官网：https://papermc.io/downloads/velocity

核心配置文件：velocity.toml

Velocity使用TOML格式配置文件，语法更现代化。

完整配置示例

# Velocity配置文件 v2.0
# 文档: https://docs.papermc.io/velocity/configuration

# 配置版本（不要修改）

# 绑定地址和端口

forwarding-secret = "your-secret-key-here-change-this"

# 公告系统
[announcements]
# 是否显示公告

[compression]
# 压缩阈值（-1=使用Minecraft默认，0=禁用）

level = -1

# Query协议设置
[query]

# Query端口
port = 25577
# 是否显示插件信息

# 服务器列表

# 大厅服务器

# 生存服务器

# 尝试连接的服务器顺序
try = [
    "lobby"
]

# 强制主机（域名绑定）
[forced-hosts]
"lobby.example.com" = [
    "lobby"
]
"pvp.example.com" = [
    "minigames"
]

# 高级设置
[advanced]
# 压缩级别
compression-level = -1
# 登录超时
login-ratelimit = 3000
# 连接超时
connection-timeout = 5000
# 读取超时
read-timeout = 30000
# 是否启用HAProxy协议
haproxy-protocol = false
# TCP快速打开
tcp-fast-open = false
# BungeeCord插件消息通道
bungee-plugin-message-channel = true
# 显示ping请求
show-ping-requests = false
# 是否在代理后启动
proxy-protocol = false
# 网络ID

[metrics]
# 是否启用bStats统计
enabled = true
# 统计ID（自动生成）
id = "random-uuid-here"
# 日志失败
log-failure = false

Velocity的现代化转发

Velocity的现代化转发模式比BungeeCord的IP转发更安全。

配置步骤

  1. 代理端配置（velocity.toml）

    # 启用现代模式
player-info-forwarding-mode = "modern"

# 设置密钥（必须是强密码）
forwarding-secret = "your-very-secure-secret-key"

  2. 子服端配置（Paper）

    编辑 config/paper-global.yml（Paper 1.19+）或 paper.yml（旧版本）：

    proxies:
  velocity:
    enabled: true
    online-mode: true
    secret: "your-very-secure-secret-key"  # 必须与代理端一致

    编辑 server.properties：

    online-mode=false

  3. 生成安全密钥

    # Linux生成随机密钥
openssl rand -base64 32

# 或使用Velocity自带工具
java -jar velocity.jar generateSecret

Velocity启动脚本

Linux启动脚本

start.sh
#!/bin/bash
screen -dmS velocity java -Xms512M -Xmx1G -XX:+UseG1GC -XX:G1HeapRegionSize=4M -XX:+UnlockExperimentalVMOptions -XX:+ParallelRefProcEnabled -XX:+AlwaysPreTouch -jar velocity.jar
echo "Velocity已启动"

准备工作

硬件需求

首先得确保你的服务器配置满足以下配置要求：

最低配置

  • CPU：2核心
  • 内存：4GB RAM
  • 硬盘：20GB SSD
  • 网络：10Mbps上传

推荐配置

  • CPU：4核心+

BungeeCord插件开发基础 拓展

还想开发插件？也是可以的，如果你想为BungeeCord开发自定义插件，这里倒是有一个简单示例。

  1. 开发环境搭建

  2. 创建插件主类

    package com.example.myplugin;

import net.md_5.bungee.api.plugin.Plugin;
import net.md_5.bungee.api.ChatColor;
import net.md_5.bungee.api.ProxyServer;

public class MyPlugin extends Plugin {
    
    @Override
    public void onEnable() {
        // 插件启用时执行
        getLogger().info(ChatColor.GREEN + "插件已启动！");
        
        // 注册命令
        getProxy().getPluginManager().registerCommand(this, new MyCommand());
    }
    
    @Override
    public void onDisable() {
        // 插件禁用时执行
        getLogger().info(ChatColor.RED + "插件已关闭！");
    }
}

  3. 创建plugin.yml

    plugin.yml
    name: MyPlugin

  4. 跨服传送示例

    import net.md_5.bungee.api.CommandSender;
import net.md_5.bungee.api.connection.ProxiedPlayer;
import net.md_5.bungee.api.plugin.Command;

public class MyCommand extends Command {
    
    public MyCommand() {
        super("hub");  // 命令名称
    }
    
    @Override
    public void execute(CommandSender sender, String[] args) {
        if (!(sender instanceof ProxiedPlayer)) {
            sender.sendMessage("此命令只能由玩家执行！");
            return;
        }
        
        ProxiedPlayer player = (ProxiedPlayer) sender;
        
        // 获取目标服务器
        ServerInfo lobby = ProxyServer.getInstance().getServerInfo("lobby");
        
        if (lobby == null) {
            player.sendMessage("§c错误：找不到大厅服务器！");
            return;
        }
        
        // 传送玩家
        player.connect(lobby);
        player.sendMessage("§a正在传送到大厅...");
    }
}

常见插件兼容性问题

1. 子服插件无法识别真实玩家数据

范例：某些插件可能无法正确读取玩家信息，大概率是因为配置错误而出的问题。

解决方案：确保子服已正确配置：

  • server.properties 中 online-mode=false
  • spigot.yml 中 bungeecord: true
  • Paper配置中启用代理支持

2. 经济插件不跨服同步

范例：玩家在不同的子服务器的金币余额不同步，即数据不同步。

解决方案：使用MySQL数据库存储经济数据 or 使用支持BungeeCord的经济插件（如PlayerPoints、TokenManager）

3. 权限在不同服务器不一致

范例：玩家在生存服有管理员权限，但在小游戏服没有。

解决方案：使用LuckPerms并配置MySQL同步 or 在所有服务器安装LuckPerms并连接同一数据库

常见问题解答（FAQ）

问题1：玩家无法连接到代理服务器

症状：
“无法连接到服务器”；“连接超时”

排查步骤：

  1. 检查BungeeCord是否正在运行

    # Linux
ps aux | grep BungeeCord

# 查看端口监听

# Windows - 在防火墙设置中添加规则

  3. 检查配置文件中的bind地址

    config.yml
    host: 0.0.0.0:25577  # 确保是0.0.0.0，不是127.0.0.1

症状：
使用 /server 命令后断开连接；卡在“正在连接”界面

  2. 检查子服配置

    # server.properties

  3. 检查网络可达性

    # 从BungeeCord服务器ping子服

“Failed to verify username”，正版玩家无法进入

安全注意事项 重要

1. IP转发安全

如果配置不当，某些恶意玩家就可能伪造IP地址。你可以这样配置来避免这种风险：

BungeeCord模式：

# BungeeCord config.yml
ip_forward: true
# Paper配置
proxies:
  bungee-cord:
    online-mode: true

并且必须：防火墙阻止外部直接访问子服端口！

# Linux防火墙规则
# 只允许代理服务器IP访问子服端口
sudo ufw allow from 10.0.0.1 to any port 25565
sudo ufw deny 25565

Velocity模式：

使用modern转发模式更安全：

# velocity.toml
player-info-forwarding-mode = "modern"
forwarding-secret = "strong-random-secret-key"

2. 防火墙配置

你需要明确的基本原则是👇👇👇

基本原则

只开放代理服务器端口（如25577），子服务器端口不对外开放，以及记得用使用白名单IP访问

Linux（UFW）配置

# 默认拒绝所有连接
sudo ufw default deny incoming
sudo ufw default allow outgoing

# 允许SSH
sudo ufw allow ssh

# 允许BungeeCord端口
sudo ufw allow 25577/tcp

# 拒绝子服端口（外部访问）
sudo ufw deny 25565/tcp
sudo ufw deny 25566/tcp

# 如果代理和子服在不同机器，允许特定IP
sudo ufw allow from 代理服务器IP to any port 25565

# 启用防火墙
sudo ufw enable

Windows防火墙配置

  1. 打开“高级安全Windows Defender防火墙”
  2. 新建入站规则
  3. 规则类型：端口
  4. 协议：TCP，端口：25577
  5. 操作：允许连接
  6. 配置文件：全部勾选
  7. 名称：BungeeCord

对子服端口创建阻止规则。

3. DDoS防护

对于DDoS，我们没有太多的办法去防护，因为DDoS的本质就是利用大量的“肉鸽”去制造大量的连接请求，让服务器崩溃。但不过你仍然可以通过以下方式来在一定程度上反制DDoS：

可以在BungeeCord config.yml中配置连接限制：

config.yml
# BungeeCord config.yml
connection_throttle: 4000  # 限制同一IP的连接频率
connection_throttle_limit: 3  # 4秒内最多3次连接

当然，也可以使用TCPShield。TCPShield是专门为Minecraft设计的DDoS防护服务。

  1. 注册TCPShield账号
  2. 添加你的服务器
  3. 配置BungeeCord使用TCPShield后端
  4. 更新DNS记录指向TCPShield

4. 定期备份

配置一个自动备份脚本：

#!/bin/bash
# backup.sh

BACKUP_DIR="/backups/minecraft"
DATE=$(date +%Y-%m-%d_%H-%M-%S)

# 备份BungeeCord
tar -czf $BACKUP_DIR/bungeecord_$DATE.tar.gz /path/to/bungeecord

# 备份子服
tar -czf $BACKUP_DIR/lobby_$DATE.tar.gz /path/to/lobby
tar -czf $BACKUP_DIR/survival_$DATE.tar.gz /path/to/survival

# 删除7天前的备份
find $BACKUP_DIR -type f -mtime +7 -delete

echo "备份完成: $DATE"

添加到crontab（每天凌晨3点执行）：

crontab -e
# 添加这行
0 3 * * * /path/to/backup.sh

5. 最小权限原则

最重要的是，不要以root用户运行服务器！！！
还有别忘了为每个服务创建专用用户，限制文件系统访问权限

# 创建专用用户
sudo useradd -m -s /bin/bash minecraft

# 修改文件所有者
sudo chown -R minecraft:minecraft /path/to/servers

# 限制权限
chmod 750 /path/to/servers

7. 日志监控

启用并定期检查日志文件：

# config.yml
log_commands: true  # 记录玩家命令
log_pings: false    # 不记录ping（减少日志量）

使用日志分析工具检测异常这三个方面：

  • 异常大量的连接尝试
  • 重复的错误信息
  • 可疑的命令执行
